# Context-Aware Food Recommendation Agent (Course Project Prototype)

## Overview
This project explores how smartphone app usage behavior can be used to infer high-level user context states, then connect those states to dish-level food recommendations.

The core idea is:
1. Analyze app usage sessions from LSApp.
2. Discover behavioral patterns with unsupervised learning.
3. Interpret clusters cautiously as possible user states.
4. Use inferred state to recommend dishes from a GrubHub-like menu dataset.

## Dataset
Primary dataset: LSApp
- Repository: https://github.com/aliannejadi/LSApp
- Domain: smartphone app usage logs

Important constraint:
- LSApp does not provide mood labels.
- The first stage should not train a supervised emotion classifier.
- The correct first step is unsupervised clustering of behavior patterns.

## Project Goals
1. Build a reproducible data pipeline for LSApp event/session analysis.
2. Extract meaningful session-level behavioral features.
3. Cluster sessions into interpretable usage modes.
4. Produce clear summaries and visualizations for each cluster.
5. Prepare a clean interface for a later dish recommendation stage.

## Out of Scope (Early Phase)
1. Claiming exact emotions from app usage.
2. Building a fully deployed production recommender.
3. Personal diagnosis or psychological profiling.

## Column Explanations

| Column | Type | Description |
|---|---|---|
| `interaction_id` | integer | Interaction segment ID generated from event stream rules (new ID when app/user changes or when opened-event gap exceeds threshold). |
| `user_id` | numeric/string | User identifier from LSApp logs (source column `lsapp.tsv` renamed to `user_id`). |
| `session_id` | integer | Session segment ID generated by 5-minute opened-event gap rule plus user boundary rule. |
| `timestamp` | datetime | Timestamp of the first event in this interaction (because interaction table keeps first row per interaction). |
| `app_name` | string | App associated with the interaction (taken from the first row of the interaction). |
| `event_type` | string | Event type of the first interaction row (for example `Opened`, `Closed`). |
| `time_dff` | timedelta | Row-to-row time difference from raw sorted stream (`current timestamp - previous timestamp`). This is not interaction duration. |
| `open_time` | datetime | Start time of the interaction (minimum timestamp within the interaction). |
| `close_time` | datetime | End time of the interaction (maximum timestamp within the interaction). |
| `duration(s)` | float (seconds) | Interaction duration in seconds, computed as `close_time - open_time`. |
| `app_category` | string | Category label merged from `output/UniqueAppNames/app_category_mapping_87.txt` by exact match on `app_name`. |

## Planned Development Stages

### Stage 1: Data Loading, Cleaning, and Clustering
Target tasks:
1. Load LSApp data from local files or download source.
2. Inspect raw fields, especially:
	- user_id
	- session_id
	- timestamp
	- app_name
	- event_type
3. Clean event logs:
	- remove obviously invalid rows
	- handle broken/corrupt events if present
	- parse and normalize timestamps
	- sort by user/session/time
4. Create session-level features, such as:
	- session duration
	- number of app opens
	- number of app switches
	- number of unique apps
	- app diversity (entropy)
	- time-of-day signal
	- weekday/weekend indicator
	- dominant app/category in session
	- category ratios (social, entertainment, productivity, utility) when mapping is possible
5. Standardize numerical features.
6. Cluster sessions:
	- start with KMeans
	- optionally compare GMM and DBSCAN/HDBSCAN
7. Evaluate clustering quality:
	- silhouette score
	- Davies-Bouldin score
	- cluster size distribution
	- 2D projection plots (PCA or UMAP)

Expected Stage 1 artifacts:
1. Processed feature table (CSV)
2. Cluster label table (CSV)
3. Cluster visualization figures
4. Basic cluster summary report

## Output Data Dictionary (Interaction-Level File)

Current notebook output file:
- `data/with_time_diff/lsapp_interactions_with_time_diff_filtered.tsv`

This file is interaction-level (one row per `interaction_id`) and is filtered to keep records where `duration(s) >= 3`.

It also includes app category labels in the same file (no separate category output file).

### Category Mapping Inputs

- `output/UniqueAppNames/unique_app_names.txt`
	- deduplicated app list extracted from LSApp.
- `output/UniqueAppNames/app_category_mapping_87.txt`
	- tab-separated mapping file with exactly two columns: `app_name`, `app_category`.

### Category Merge Rules

1. Merge is exact-match on `app_name`.
2. Do not infer categories in code.
3. If any `app_name` has missing `app_category`, the notebook raises an error and stops saving.
4. When no missing mappings are found, the filtered output file is saved with `app_category` included.


### Stage 2: Cluster Interpretation (Cautious)
Interpret clusters as behavioral usage states rather than fixed emotions.

Example state descriptions:
1. focused/task-oriented usage
2. fragmented/rapid-switching usage
3. social-checking usage
4. entertainment-heavy usage
5. late-night/restless usage

Interpretation language should remain cautious, for example:
1. may suggest
2. could be associated with
3. might reflect

### Stage 3: Dish Recommendation Layer
After inferring user behavioral state, connect state to a menu recommendation use case.

Planned behavior:
1. Use a GrubHub-like menu dataset (or mock local menu data).
2. Recommend specific dishes, not only restaurants.
3. Rank dishes using state-aware rules/signals.

Example mapping ideas:
1. fragmented/restless/late-night state -> comfort food, warm dishes, low-effort meals
2. focused/task-oriented state -> quick meals, easy-to-eat options, delivery-efficient dishes
3. entertainment-heavy state -> snacks, desserts, shareable/fun foods
4. social-checking state -> popular/trendy dishes, group-friendly items

### Notes for Team Usage

1. `time_dff` and `duration(s)` are different concepts:
	- `time_dff`: event-level adjacent delta
	- `duration(s)`: interaction-level total span
2. If you need pure interaction features for modeling, prefer `interaction_id`, `user_id`, `session_id`, `app_name`, `open_time`, `close_time`, and `duration(s)`.
3. This output is already filtered with `duration(s) >= 3`, so very short interactions have been removed.
